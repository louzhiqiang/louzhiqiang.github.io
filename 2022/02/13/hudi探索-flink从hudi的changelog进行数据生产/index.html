hudi探索–flink从hudi的changelog进行数据生产

在数据生产中，我们都希望流式数据生产可以实现端到端的全流程，也就是希望从A到B到C的生成，但是中间不能再增加个什么消息中间件之类的，否则这套服务就会变得臃肿。而flink cdc的诞生，从开始就是以变化事件来触发变更作为场景切入，完成了oltp到数仓的链接，但是数仓内部多层次的生产还没有完成，这部分还是通过消息中间件来链接，而这好像跳出了框架，也缩小了flink-cdc的愿景。然而如果仓库里的数据本身就是支持cdc格式的话，那么一切是不是就可以直接顺下来了，全部都靠flink的内部机制 + 一个支持cdc格式的存储，这事儿是不是就能统一解决了呢？

事实也是这样的，hudi底层本身就是这么存储的，因为hudi从创建之初就在挑战大数据仓库跟oltp技术栈的融合，而这样的方式就是让我们有更多的思考入口可以接纳这套架构，从底层基础知识的构建上，也越来越趋向于统一，在接受门槛上，相对来说更低了。这一点其实对于一个技术的推广更加方便，也更容易，因为大家的学习成本降低了，学习成就感更强了。

IncrementalInputSplits的官方注释：

Utilities to generate incremental input splits MergeOnReadInputSplit. The input splits are used for streaming and incremental read.
How to generate the input splits:
1、first fetch all the commit metadata for the incremental instants;（首先获取commit元数据）
2、resolve the incremental commit file paths;（解析增量提交的路径）

3、filter the full file paths by required partitions;（根据提供的partition，进行filter）
4、use the file paths from #step 3 as the back-up of the filesystem view.(从第三步过滤数据的所有文件路径作为数据读取的视图)

构建IncrementalInputSplits的时候有个比较重要的配置：

read.streaming.skip_compaction：是否跳过流读取的压缩。在两种情况下，可以使用该选项来避免读取重复:1、确定消费者的读取速度快于压缩速度，这个时候可以看一下读取周期或者说时间跨度。2、启用了更改日志模式，此选项是保持数据完整性的解决方案。

这里要重点关注一下：

1、splits 是一个queue，也就是每次数据处理都是拿一个split。

2、用format打开这个split，也就是加载的概念。open主要是把数据生成一个iterator，会根据不同的类型生成不同类型的迭代器。hudi主要就是利用这个方式来兼容不同的数据组织方式，然后处理代码就可以变得统一。

所有的迭代器 都是实现的接口 RecordIterator 。

3、生成了迭代器之后，还要把split跳转到需要的地方，因为一次读取数据最多 MINI_BATCH_SIZE ， 所以对于大于 MINI_BATCH_SIZE 的split，是会被多次读取才能消耗完。这个时候就需要去skip到指定位置，才能继续消耗后边。

这部分代码主要的逻辑就是进行parquet数据和log数据的merge。reader就是parquet数据的读取，是逐行读取的。读取的是基础数据。然后log数据的读取是scanner，也是逐行读取，他会把所有文件都读取一遍，然后按照分区和文件id，进行分组，在遍历parquet文件的时候，会进行匹配，然后把同一个slice的数据进行merge，这样就相当于组织了一个全量数据，数据相对于当前的instantRange来说，也就是最新的数据了。

这里的数据组织方式，往通用逻辑去考虑也是一致的。这里的通用逻辑可以归纳到数据库本身对于变化数据的处理。不过也要冷静去想一下hudi这种基于文件的组织方式的缺点是啥？在什么极端条件下会有哪些风险。

主题思路就是monitor通过format把服务出来的数据组合成一个个的split，而一个个split就是fileSlice的概念。然后会有orerator算子来消费这些split，通过指定task数量的任务进行消费，转化为RowData。
