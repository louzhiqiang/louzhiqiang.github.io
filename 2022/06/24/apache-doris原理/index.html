apache doris原理

最近在多维分析的技术峰会上，doris的身影比较频繁，在各个大厂都有出现，所以他的适用场景以及技术原理，需要了解一下。

出处

Doris（原百度 Palo）是一款基于大规模并行处理技术的分布式 SQL 数据库，由百度在 2017 年开源，2018 年 8 月进入 Apache 孵化器。2022年6月正式毕业成为顶级项目。

Doris 整个的定位

  • MPP 架构的关系型分析数据库

  • PB 级别大数据集，秒级/毫秒级查询

  • 主要用于多维分析和报表查询

架构

  • 以数据存储的角度观之，FE 存储、维护集群元数据；BE 存储物理数据。

  • 以查询处理的角度观之, FE 节点接收、解析查询请求，规划查询计划，调度查询执行，返回查询结果；BE 节点依据 FE 生成的物理计划，分布式地执行查询。

FE,BE 都可线性扩展。

FE：主要负责查询的编译，分发和元数据管理

FE 主要有有三个角色，一个是 leader，一个是 follower，还有一个 observer。leader 跟 follower，主要是用来达到元数据的高可用，保证单节点宕机的情况下，元数据能够实时地在线恢复，而不影响整个服务。

右边 observer 只是用来扩展查询节点，就是说如果在发现集群压力非常大的情况下，需要去扩展整个查询的能力，那么可以加 observer 的节点。observer 不参与任何的写入，只参与读取。

BE是存储数据的，支持多副本配置。不依赖其他外部组件，所以部署简单易用。

doris的存储可以支持分层存储，也就是热数据可以用更高效的硬件支持，冷数据可以用更低成本的硬件去支持。

向量化执行

利用向量化执行，提高查询效率。

后边专门调研一下已经实现了向量化执行的这些引擎，比如clickhouse、druid的新版本。

索引

doris目前主要是两种索引：稀疏索引以及前缀索引。

后者面对一些灵活查询的场景，覆盖不够全面，总会出现无法覆盖的查询分析。

物化视图

在 Doris 里，物化视图和底表数据是一致的，就是对于物化视图和底表，数据在写入时是一个事务，是同步操作，可以保证一致性。

物化视图比较多的话，就会出现写放大的问题，比较难管理。

联邦查询

现在很多大厂对于联邦查询的需求比较明显，doris目前在这个方向上的扩展也比较突出，比如doris外联RDBMS、外联es、外联hdfs上的一些数据表，比如基于orc的表，基于iceberg的表以及基于hudi存储的书，都已经有用户相继实现了对应的一些封装和外联使用。

Doris on Elasticsearch

join

doris 不仅自适应Broadcast/Shuffle Join 方式，也可手动进行 Colocation Join ，也能在这些join 基础上自适应的进行两阶段聚合。

系统默认是 shuffle join，也被称作 Partitioned Join。即将小表和大表都按照 Join 的 key 进行 Hash，然后进行分布式的 Join。这个对内存的消耗就会分摊到集群的所有计算节点上。

Doris会自动尝试进行 Broadcast Join，如果预估小表过大则会自动切换至 Shuffle Join。注意，如果此时显式指定了 Broadcast Join 也会自动切换至 Shuffle Join。

运行时会对数据进行采样，来判断是否进行两阶段聚合

JOIN 优化 Colocation Join

  • Colocation Group（CG）：一个 CG 中会包含一张及以上的 Table。在同一个 Group 内的 Table 有着相同的 Colocation Group Schema，并且有着相同的数据分片分布。
  • Colocation Group Schema（CGS）：用于描述一个 CG 中的 Table，和 Colocation 相关的通用 Schema 信息。包括分桶列类型，分桶数以及副本数等。
原理

Colocation Join 功能，是将一组拥有相同 CGS 的 Table 组成一个 CG。并保证这些 Table 对应的数据分片会落在同一个 BE 节点上。使得当 CG 内的表进行分桶列上的 Join 操作时，可以通过直接进行本地数据 Join，减少数据在节点间的传输耗时。为了使得 Table 能够有相同的数据分布，同一 CG 内的 Table 必须保证以下属性相同：

  • 分桶列和分桶数
    分桶列，即在建表语句中 DISTRIBUTED BY HASH(col1, col2, …) 中指定的列。分桶列决定了一张表的数据通过哪些列的值进行 Hash 划分到不同的 Tablet 中。同一 CG 内的 Table 必须保证分桶列的类型和数量完全一致，并且桶数一致，才能保证多张表的数据分片能够一一对应的进行分布控制。

  • 副本数
    同一个 CG 内所有表的所有分区（Partition）的副本数必须一致。如果不一致，可能出现某一个 Tablet 的某一个副本，在同一个 BE 上没有其他的表分片的副本对应。

同一个 CG 内的表，分区的个数、范围以及分区列的类型不要求一致。

查看查询计划，如果 Colocation Join 生效，则 Hash Join 节点会显示 colocate: true。

colocation 跟 hologres 的 tableGroup的概念有些贴边，主要思想就是期望通过一定的设置可以敢于数据分布，让有业务相关改变的数据按照一定规则分布到一个计算节点，这样会导致某些高频查询效率高很多。

不过这里还要求分桶数也一致，所以这里对于相关性影响也是更加复杂的。

向量化执行

引入向量化计算的原因：

  • 虚函数调用，无法内联，编译器获取的信息就少了，也就导致编译器优化的空间就减小了。

  • 充分利用缓存（寄存器）

  • 减少分支跳转

  • 提高分支预测的准确率
