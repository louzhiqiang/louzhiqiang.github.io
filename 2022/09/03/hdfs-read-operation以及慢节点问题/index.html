hdfs读取的代码如下：

  • 客户端通过调用 FileSystem 对象上的 open() 打开它希望读取的文件，对于 HDFS 来说的话，它是 DistributedFileSystem 的一个实例。
  • DistributedFileSystem 使用 RPC 访问 namenode 来确定文件中前几个块的块的位置。 对于每个块，namenode 返回具有该块副本的 datanode 的地址，并且 datanode 根据它们与客户端的接近程度进行排序。
  • DistributedFileSystem 将 FSDataInputStream 返回给客户端，以便从中读取数据。 因此，FSDataInputStream 包装管理 datanode 和 namenode I/O 的 DFSInputStream。客户端在流上调用 read()，DFSInputStream 存储了 datanode 地址，然后连接到文件中第一个块的最近的 datanode 来读取。
  • 数据从 datanode 流回客户端，因此客户端可以在流上重复调用 read()。当块结束时，DFSInputStream 将关闭与 datanode 的连接，然后找到下一个块的最佳 datanode。
  • 如果 DFSInputStream 在与 datanode 通信时遇到错误，它将尝试该块的下一个最接近的一个。它还会记住失败的数据节点，这样它就不会为了以后的块而不必要地重试它们。DFSInputStream 还验证从 datanode 传输给它的数据的校验和。 如果它找到损坏的块，它会在 DFSInputStream 尝试从另一个 datanode 读取块的副本之前将其报告给 namenode。
  • 当客户端读完数据后，它会调用流上的 close()。

DFSInputStream.read

我们执行 read 的入口函数是 DFSInputStream 里的 read 函数，注意，DFSInputStream 里有多个 read 函数，你可以理解为下面这个是个顺序读的 read 函数，并且它采用Byte数组来读。

首先它会将 buf 封装到 ByteArrayStrategy 里，ByteArrayStrategy 的结构比较简单，主要是封装了 buf，并且利用 BlockReader 具体执行 read 的 doRead 函数。

readWithStrategy

readWithStrategy就是具体执行读的逻辑了，我们先上整个函数的代码。

大体的逻辑很容易理解，retries 设为 2，意味着我们有两次重试的机会，在 read 时抛异常，相应的会在 catch 里将当前使用的datanode 加入 dead node 的 list 里，避免下次再选取这个 datanode 来进行读，最后在 finally 里做一次 checksum failure 的检查。
我们根据每次读入的字节数来更新pos，从而读完整个文件，接下来我们分析一下几个重点函数。

blockSeekTo

我们要根据 pos 来拿到需要读取的 datanode 来更新 currentNode，通过 blockSeekTo 函数完成。

这个函数的逻辑也很容易理解，在一个死循环里不停的执行 getBlockAt 和 chooseDataNode 直到找到一个能用的 datanode，然后根据 datanode 构造新的 blockReader，返回 DataNodeInfo，在这个过程里同样会判断死节点并加到 dead nodes 里。

我们分别看一下 getBlockAt 和 chooseDataNode 这两个函数。

getBlockAt

getBlockAt 根据 offset 得到具体的 LocatedBlock。

这里主要利用 locatedBlocks 去拿到我们需要的 block，locatedBlocks 是通过 DFSClient 里拿到的一些块，它们是 cache 的。如果没有找到我们的块，我们会通知 DFSClient 去拿新的一批块。
还需要注意的是 blockEnd 在这里更新了，更新为我们拿到的 block 结束的位置。

chooseDataNode

一个 block 会有多个副本，它们存在不同的 datanode 上，chooseDataNode 会根据 block 拿到具体的最合适的 datanode。

getBestNodeDNAddrPair

它会遍历 block 的所有副本，它们存在不同的 datanode 上，这里 best 是指之前 namenode 返回给我们 blocks 的时候，已经根据一些条件对所有副本进行了排序，我们根据排序优先级选择第一个还没有试过的副本，之前用过但出问题的 datanode 会被加到 ignored nodes 里。

因为上一步传入的 ignoredNodes 为空，这里的 ignoredNodes 基本是个摆设，主要靠 deadNodes 来过滤掉一些坏节点。
　　如果你看懂这里的逻辑，也就明白了为啥对于慢节点问题，就算重启一个 speculative task，依然会出现慢节点问题，因为它们读的同一个副本，因为很可能 namenode 提供的副本排序并没有变，所以它们读的依然是第一个不是坏节点的节点，只不过它是慢节点。

readBuffer

如果在读的过程中抛异常，对于第一个副本它会重试一次相同的副本，如果还是出错，会将当前 datanode 加入 dead node，并且采用新的 datanode 上的副本，直到成功读取或者所有的副本都出错抛异常跳出循环。
它具体执行读的逻辑我们前面的 ByteArrayStrategy 里的 doRead 函数里讲过，里面调用了 blockReader.read 方法，这里的 blockReader 大多 RemoteBlockReader2 实现。

reportCheckSumFailure

这主要进行的文件校验的工作，Hadoop为了保证数据的一致性，会对文件生成相应的校验文件，并在读写的时候进行校验，确保数据的准确性，如果出现了错误，一些情况下需要汇报给 namenode。

我们看到有两种情况会给 namenode 汇报：

  • 尝试了多个副本，其中至少一个成功了，其他失败的副本会被汇报给 namenode。
  • 如果所有副本都失败了，只有当副本数为 1 时才会被汇报给 namenode，否则不汇报。

Datanode 在把数据实际存储之前会验证数据的校验和。client 通过 pipeline 把数据写入 datanode，最后一个 datanode 会负责检查校验和。当 client 从 datanode 读取数据时，也会检查校验和。把真实数据的校验和 datanode 上的校验进行比较。
　　每个 datanode 都保存有一个 checksum 验证的持久化日志，日志中有当前 datanode 每个 block 最后的更新时间。当 client 成功验证了一个 block，它会告诉 datanode，之后 datanode 会更新它的日志。保存这些信息有助于检测坏磁盘。
　　除了会在 client 读取数据时验证 block，每个 datanode 还会在后台运行一个 DataBlockScanner 线程来周期性验证所有存储在 datanode 上的 block，这用来防止物理存储媒介上出现”位衰减”。
　　因为 HDFS 保存有同一个 block 的多个备份,所以它可以用好的副本来替换损坏的数据副本。如果某个 client 在读取数据时检测到数据错误，在抛出 ChecksumException 后，它会向 namenode 上报信息告知具体的 bad block 还有 datanode。namenode 会把指定的 block 标记为 corrupt，之后的 client 就不会再被定位到此 block，此 block 也不会再被复制到其它 datanode。之后 namenode 会调度一个此 block 的正常副本，以保证负载平衡。这之后，损坏的 block 副本会被删除。
　　可以在对 FileSystem 调用 open() 之前调用 setVerifyChecksum() 来禁止校验和检测，也可以通过在 shell 中执行 get -copyToLocal 命令时指定 -ignoreCrc 选项做到。

总结

我们通过阅读 read operation 的源码，了解了 HDFS 读文件的整个过程，以及它的一些重试以及容错功能，并且知道了为什么对于读慢节点问题，如果重启 speculative task，依然会很慢，主要问题是因为它读的依然是同一个节点。
