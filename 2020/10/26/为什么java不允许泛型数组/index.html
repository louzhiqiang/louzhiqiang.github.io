为什么java不允许泛型数组

Posted on 2020-10-26 | In java

背景

该句编程环境会提示错误，连编译都不能通过

为什么

为什么会有泛型

未引入泛型之前：

两个问题:1、当获取一个值时必须进行强制类型转换。
2、没有错误检查，可以向数组列表中添加任何类的对象。也就是add的时候不报错，因为所有对象是object的子类，数组是允许往当前数组中添加当前数组声明的类的子类（这个被称作是“协变(covariant)”）。但是在运行的时候，如果有转换操作，就会产生ClassCastException的异常。 而这样的操作实际上很多的，那么出错的风险就大大增加，安全性上没法对添加的元素去限制，安全性上也需要考量。

所以，java引入了泛型。

目的：1、将类型检查工作提前到编译时期。2、将类型强转（cast）工作交给编译器。

这样你就可以 在编译时期就获得类型转换异常以及去掉源码中的类型强转代码。

为什么java的泛型被设计成可擦除的

参考帖子：入口

总结：java其实在很早就有过泛型的开发，在其中的某一个版本上就实现过基于jvm底层的泛型机制，这个版本是pizza，但是这个项目后来消亡了，相关开发人员就投入了另一个java分支的项目，又扩展了通配符等特性之后，就构成了现在java泛型的雏形，所以java从一开始对泛型的开发投入就是有限的，支持并不到位。

而且鉴于java的用户团体的庞大，考虑大版本升级会带来程序员用户的流失，所以Java所强调的兼容性，是“二进制向后兼容性”，例如，一个在java1.2、java1.4版本上可以正常运行的class文件，放在一个java5、6、7、8的JRE（JVM和标准库）上仍然可以正常运行。而class文件这里就是java程序的二进制表现。

“二进制兼容性”并不等价于“源码兼容性”（source compatibility）

Java从来都不支持高版本的Java编译生成的Class文件在低版本的JRE上运行。如果有人尝试这样做，就会得到UnsupportedClassVersionError—-class文件开头所记录的版本号信息的一个主要用途就是这个。

以上这段代码，在做泛型化的时候，务必要考虑的，因为实现泛型之后，这段代码也要能够在新的JRE上运行。所以有两个方向的思路：
1、需要泛型化的类型（主要是collection），以前有的保持不变，然后平行地加一套泛型化版本的新类型。也就是通过新加类型的方式来兼容。
2、直接把已有的类型泛型化，让所有需要泛型化的已有类型都原地泛型化，不添加任何平行于已有类型的泛型版。

Java选择了第二个方案。.net选择了第一个方案。

.net这个选择很明白：实际应用的代码量有限（用户没有那么庞大）；可以主动变更，因为都在微软的掌控之下。

java就不一样，那时候java已经有大量用户，发布了新功能之后，如果用户需要改动大量源码才能使用上泛型，那势必会被大量吐槽。针对于方案一的方式，java早起就经历过那样的方案实例，就是关于java collections frameWork的新版本，在java 1.1 –> 1.2的时候，共存了两个版本的容器（Collections），这个当时的用户吐槽疯了。所以后来java就选了第二个方案。

但是为了兼容ArrayList的写法,怎么兼容呢?在原地泛型化之后,现在这个类型变成了ArrayList,为了让ArrayList也可以直接使用，所以就引出了一个”raw type“的概念—-一个类型虽然被泛型化了，但还可以把它当作非泛型化的类型用。

注意，这三者并不等价。这个可以在effecttive java中有实际case的说明。

要支持raw type，最直接的办法就是通过擦除法来实现泛型—-编译时类型确实是泛型的，但是编译之后，泛型信息要被擦除，最后的结果就是raw type（加上一些额外的类型检查，如果有wildcard bounds的话还会带上bounds信息）。

所以，Java不允许创建泛型数组。

那么应该怎么使用泛型呢？（参考effective java 第五章）

1、不要使用原生态类型。这个原生态主要是容器一类的原生态类型，比如Collections、List等。原生态类型是不安全的，多考虑通配符类型这样的操作。
2、列表优于数组。数组是协变且可以具体化的，泛型化是不可变的且可以被擦除的。数组提供了运行时的类型安全，但是没有编译时的类型安全；反之，对于泛型也是一样。两者不能混用，优先列表。因为java数组天然协变性的缺陷。
3、优先考虑泛型方法。
4、谨慎并用泛型和可变参数：可变参数的实现是数组，将值保存在泛型可变参数数组参数中，是不安全的，破坏了泛型系统的基本保证。
5、优先考虑类型安全的异构容器。

收益

使用泛型的好处：

通用一点说就是，在编译的时候检查类型安全，并能捕捉类型不匹配的错误，并且所有的强制转换都是隐式的和自动的，提高代码的重用率。也就是模板化和规范化，模板化就是不用写很多各种转换或者各种对比或者转换，而规范化，就是在使用泛型的时候，泛型会提供校验。

总结

尽管泛型有很多的限制，但是泛型加入Java语言所带来的价值是非常庞大的。如今很难去想象如果Java没有泛型支持将会是什么样的一幅景象。应该用泛型而不是原始类型（Collection代替Collection, Callable<T&gt 代替 Callable, …）和Object来保证类型安全，在约定和算法上定义明确的类型约束，大大简化代码维护和重构。

然而，请一定要注意当前Java中泛型实现的约束，类型擦除和我们所熟知的基本类型的隐式拆箱和装箱。泛型并不是解决所有可能遇到的问题的银弹并且任何东西都无法取代精心的设计和深思熟虑。

最好的方式就是看一些泛型使用的真实的实例，领略泛型的合理使用给开发人员带来的工作上的便利。
