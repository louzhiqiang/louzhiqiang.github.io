一个presto查询计划的分析

发表于 2020-10-21 | 分类于 大数据 ， presto

sql如下： select * from (select filed_a,field_b from table_a where field_a=’a’ order by field_b desc) as t limit 1000

本身这个带有子查询的查询语句，我以为会是一个优化，因为按照业务需求，直接返回1000条数据就可以了。

针对这条查询的疑问是 在presto中，通过explain解析，执行计划中并 不存在排序操作，也就是排序的执行计划被优化掉了。也就是在子查询中结果集的order by是会被优化掉的。原因应该是外层数据的使用会忽略掉这个排序。如果把order by 外移，这个排序计划会在最外层。还有一点，就是order by + limit的方式，实际就是一个TOPN的一种方式。在各种引擎中，TOPN 的query都有单独的一套算法去执行，这个可以专门写个topn的查询算法解析。topn的查询跟group by的执行逻辑是有区别的。

还有一种可以让子查询排序生效的方式，那就是with子查询的方式，sql如下： with t_alias as (select * from table_a where field_a=’a’ limit 1000) select * from t_alias order by field_b desc limit 100;

这样的子查询的方式，如果在子查询中包含order by，外围查询没有的话，这个执行计划中就会包含排序的那一部分。

可见，引擎引入一个优化规则，一般会优先常见的使用方式，比如上边第一条语句的那种子查询方式，而后者这个with的写法，就没有优化。

那么以上情况应该怎么区分或者怎么使用呢？总结如下：
1、当where条件能够过滤更多的数据的时候，也就是结果集是少于limit的数量，那么上边第一种子查询的方式跟下边使用with的方式，不会有太明显的查询区别。造成区别的是order by的耗时，前者没有，后者会有。
2、where查询的结果集如果大于limit，差距越大，带有limit的查询和不带有limit的查询的耗时是越来越明显的。反之，两者耗时应该是差不多的。
3、order by的性能跟数据量直接相关，集合越大，耗时越长，所以带有order by 的语句，要考虑尽量缩小排序的集合。
